# Composer Guide Bot 产品设计

> 日期：2026-05-13
> 阶段：Phase 1 / Phase 2 候选
> 状态：设计草案

## 1. 背景

Hermes Desktop 的核心目标是让不熟悉终端的普通用户也能低门槛使用 Hermes Agent。当前 Phase 1 正在把斜杠命令图形化，但用户仍可能遇到两个问题：

- 不知道当前界面还能做什么。
- 不知道某个状态下下一步应该点哪里。

Composer Guide Bot 是一个固定在输入区附近的轻量引导角色，用低打扰的方式提示用户下一步可执行动作，帮助用户发现模型切换、排队发送、上下文压缩、目标设置、首次引导等能力。

## 2. 产品定位

Composer Guide Bot 不是新的聊天 Agent，也不是客服式浮窗。

它的定位是：

- 输入区引导层：围绕“用户准备输入/发送”的上下文给出建议。
- 本地状态提示器：优先基于前端状态、Hermes 状态、上下文进度和错误状态生成提示。
- 操作入口：把复杂斜杠命令转化为可点击动作。

它不应默认主动向 Hermes 发送消息，也不应把提示文案写入用户会话历史。

## 3. 放置位置

固定放在对话输入框左侧附近，和输入框形成一个 composer 区域。

推荐布局：

```text
[Bot]  要不要把这轮任务设成持续目标？  [设置目标] [忽略]
┌──────────────────────────────────────────────┐  [Send]
│ Message Hermes...                            │
└──────────────────────────────────────────────┘
```

设计原则：

- 不放在消息阅读区，避免遮挡长对话正文。
- 不塞进输入框内部，避免和 placeholder、光标、附件预览、人格选择器冲突。
- 和输入框保持视觉关联，让用户理解它是在提示“下一步输入/动作”。
- 在窄窗口下可退化为仅显示 Bot 图标，点击后展开一行提示。

## 4. 视觉风格

整体风格应贴合 Hermes Desktop 的 macOS 原生伴侣定位，避免网页客服插件感。

Bot 本体建议：

- 小体积，固定在输入框左上方或左侧贴边。
- 使用 Hermes 品牌识别元素，例如闪电、终端光标、小屏幕表情。
- 跟随三套主题：Claude Noir / Apple / Warp。
- 不使用大面积赛博弹窗，不遮挡主界面。

动画建议：

| 状态 | 表情 / 动画 |
|------|-------------|
| idle | 轻微呼吸、眨眼 |
| 用户输入中 | 眼睛或光标轻微跟随输入状态 |
| Agent 运行中 | 小屏幕点阵流动或思考表情 |
| 可提示动作 | 短暂亮一下，不持续跳动 |
| 出错 | 黄色提示灯或轻微皱眉 |
| 成功 | 短暂点头或亮屏 |
| 静音 | 睡眠表情 |

动画必须克制，不持续弹跳、不频繁吸引注意力。

## 5. 交互规则

提示内容只浮出一行，最多带 1 到 3 个动作按钮。

示例：

- `第一次使用？我可以带你完成 3 步配置。` `[开始引导]`
- `这次任务有点长，要不要设为持续目标？` `[设置目标] [忽略]`
- `上下文快满了，可以先压缩一下。` `[压缩]`
- `你可以继续输入，我会排到下一轮发送。`
- `当前模型可能不适合代码任务。` `[切换模型]`
- `Hermes 启动失败，要打开安装引导吗？` `[查看引导] [重试]`

交互原则：

- 默认不自动发送消息。
- 用户点击动作后，才触发 UI 操作或发送明确命令。
- 所有提示都可忽略。
- 支持静音当前提示类型。
- 支持完全关闭 Bot。
- 同一提示不应在短时间内反复出现。

## 6. 触发场景

优先支持本地可判断的场景：

| 场景 | 触发条件 | 建议动作 |
|------|----------|----------|
| 首次使用 | hermes CLI 未安装或未完成引导 | 打开三步引导 |
| 空会话 | 当前会话无消息且用户停留一段时间 | 展示开始建议 |
| Agent 运行中 | 用户继续输入 | 提示排队发送 |
| 上下文偏高 | Context 使用率超过阈值 | 触发压缩 |
| 长任务迹象 | 多轮连续任务、长输入、用户描述目标 | 建议设置目标 |
| 执行失败 | send_message 或 dashboard 启动失败 | 重试 / 查看引导 |
| 模型相关 | 模型未配置或用户询问模型 | 打开模型选择器 |

短期内不做“自由发散聊天”。如果未来要做智能建议，也应先展示建议文案，用户确认后再调用 Hermes。

## 7. 与现有功能的关系

Composer Guide Bot 可以作为多个图形化命令入口的统一提示层：

| 功能 | Bot 作用 |
|------|----------|
| feat-110 首次使用引导 | 未安装或首次启动时提示开始引导 |
| feat-101 模型选择器 | 模型缺失或不合适时提示切换 |
| feat-104 排队发送 | Agent 运行中提示继续输入会排队 |
| feat-105 目标条 | 长任务时提示设置持续目标 |
| feat-106 快照时间线 | 高风险操作前提示保存快照 |
| feat-108 后台任务面板 | 长时间任务建议后台运行 |
| feat-003/004 Context 压缩 | 上下文接近阈值时提示压缩 |

## 8. 边界与约束

必须遵守当前架构约束：

- 通信模型是一次性进程：`hermes chat -q <msg> --resume <id>`。
- Bot 的提示不进入会话历史。
- Bot 不持续持有与 Hermes 的双向连接。
- Bot 不主动执行 C 类能力，例如 `/steer`。
- 管理类入口仍委托 `hermes dashboard`，Bot 只负责提示和打开入口。

## 9. MVP 范围

第一版建议只做本地规则，不接入模型：

- 固定 Bot 位置和基础动画。
- 空会话提示。
- 首次使用引导提示。
- Agent 运行中排队提示。
- 错误恢复提示。
- Context 压缩提示。
- 关闭 / 静音机制。

不做：

- 自由聊天。
- 长篇气泡。
- 主动调用 Hermes。
- 自定义 Bot 角色市场。
- 与消息历史混排。

## 10. 成功标准

- 用户能理解 Bot 是输入区引导，不是新的聊天对象。
- 提示不会遮挡正文或干扰输入。
- 用户至少能通过 Bot 发现 3 个核心能力：首次引导、排队发送、上下文压缩。
- 所有提示都可忽略或关闭。
- 不增加 token 成本，不污染会话历史。
